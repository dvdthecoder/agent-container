## Analysis

### #150 schema reduction — measured impact

The proxy now strips per-property `description` fields from tool schemas before forwarding to vLLM, and drops all tools except `{bash, read, grep, glob}` after an edit/write call appears in context.

**Coder models** (7B and 32B) dropped from the 2026-05-05 baseline of **~32,000** prompt tokens to **~26,034–26,046** — a **−19% reduction** (~6,000 tokens per run). The proxy logs confirm the mechanism: `tools_bytes=35,193` (raw schema from opencode), `sent_bytes=32,844` (after description stripping), with additional savings on post-edit turns where the tool list shrinks from 10 to 4.

The aider baseline (2026-05-05: ~2,700 tokens) remains the floor. The opencode/aider ratio is now **9.6×** (down from **11.9×** in the baseline). Schema overhead is still dominant but the gap is narrowing.

### Qwen3 30B-A3B — thinking token context inflation

Qwen3-30B opencode prompt tokens increased from **~32,000** (2026-05-05 baseline) to **~46,149** — a **+44% increase** despite the same framework and schema reductions applied to the Coder models.

Root cause: Qwen3 is a hybrid think/no-think model. It emits `<think>` blocks (avg 1,323 completion tokens here, vs 143–170 for Coder models). Those thinking outputs become part of the conversation history and are resent as context on every subsequent turn. For a 3-turn run, ~1,300 extra tokens per prior turn compound: turn 2 includes turn 1's thinking, turn 3 includes both. The framework schema overhead is the same (~26k baseline), but the accumulated thinking context adds ~14k on top. See [lessons-learned #20](../lessons-learned.md#20-qwen3-thinking-tokens-inflate-prompt-context-on-subsequent-turns).

**Implication:** Qwen3 costs ~1.8× more per run than Coder models in opencode, not from better outputs but from thinking verbosity. The 7B and 32B Coder models remain the better cost/quality choice for straightforward coding tasks.

### Reliability

- **7B (3/3):** Fully reliable, 128s avg. Fastest and most consistent model this run.
- **32B (1/3):** Two failures on runs 2 and 3 (both ~118–180s, no PR). Completion tokens were identical (148) across both failures — likely the `end_turn` grace period race condition ([lessons-learned #19](../lessons-learned.md#19-opencode-end_turn-grace-period--file-write-race-condition)) where the edit tool call is acknowledged but the file write isn't flushed before the sandbox terminates. Reliability has regressed from the 2026-05-05 baseline (2/3 → 1/3).
- **Qwen3 (3/3):** Fully reliable despite higher cost and first-run cold-start (330s run 1 vs ~127s runs 2–3).

### Cold-start pattern

Run 1 durations are consistently elevated across all models — 125s, 307s, 330s vs 128–131s, 118–180s, 126–127s for runs 2–3. This matches the 2026-05-05 observation: the first run after a model is deployed incurs Modal container cold-start overhead (~30s GPU init + model weight paging). Subsequent runs reuse the warm container.
