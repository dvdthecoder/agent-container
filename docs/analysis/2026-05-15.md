## Why the ratio grows: structural token accounting

### aider (Chat Completions, SEARCH/REPLACE)

aider sends a **fixed-size context per request**: repo map (~1k tokens) + file content + task.
The prompt is roughly constant across turns because aider re-sends only the files being edited.
Token cost grows with **diff size** (how much code is changed), not turn count.

```
T1: repo map + mathlib.py           → ~5.6k tokens total
T2: repo map + statslib.py          → ~5.9k tokens (slightly larger file)
T3: repo map + geometry.py + shapes → ~12.4k tokens (two files, both changed)
```

### opencode (Responses API, tool calls)

opencode accumulates **full conversation history** on every turn. Each turn sends:
- All previous turns' tool calls and results (grows linearly with turns)
- All 11 tool schemas (~33KB, repeated on every request)
- The task and AGENTS.md content

Token cost grows with **turns × context size**, which compounds:

```
T1: 4 turns  × ~7.5k ctx/turn   → ~30k tokens
T2: 8 turns  × ~11k ctx/turn    → ~91k tokens
T3: 13 turns × ~18k ctx/turn    → ~243k tokens
```

The context grows because each tool result (file read, bash output, edit confirmation)
is appended to history and re-sent on subsequent turns. T3's test failures and fix cycle
added 2 extra bash+edit rounds, pushing turn count to 13 and context to 18k per request.

### The ratio diverges because both effects compound

As task complexity increases: aider grows linearly (more files), opencode grows superlinearly
(more turns × larger context per turn). At T3, opencode is spending ~241k/243k tokens on
**prompt** (context re-transmission) and only ~1.8k on actual model output.

## What it took to get opencode working on T2/T3

This session required a full proxy rewrite before valid T2/T3 opencode results were possible.
The proxy had accumulated fragile heuristics that broke as task complexity increased.

### The problem: heuristic `tool_choice` switching

The proxy was inferring "is the model done?" by counting tool calls:

| Attempt | Logic | Failure mode |
|---|---|---|
| Original | switch to `auto` after 1 edit | model exits after median, never writes variance |
| threshold=3 | stay `required` until 3 edits | model used `write` (1 call) → infinite `git push` loop |
| `write≥1 OR edit+bash≥1` | split by tool type | still broke for different task shapes |

All variants were **task-specific** — tuned for the current tier structure, not general coding tasks.

### The fix: `task_complete` explicit signal

The proxy now injects a `task_complete(summary)` tool into every request. The model calls it
when done. No counting, no guessing.

```
tool_choice = "required"  always
                  ↓
model calls task_complete()
                  ↓
tool_choice = "none"       → model writes closing text → session ends
```

**Implementation detail:** opencode cannot execute unknown tools. It stores them in conversation
history as `name='invalid'`. The proxy detects completion by checking for
`name in ("task_complete", "invalid")`.

**Why `none` not `auto`:** `tool_choice=auto` still permits tool calls. After a `required` loop,
the model defaults to calling tools even when permitted not to. `none` explicitly prohibits
any tool call, forcing a text-only closing response.

## The `edit` vs `write` discovery

Run 4 (89k tokens) revealed a second problem independent of `tool_choice`:
opencode's `edit` tool requires `oldString` and `newString` as JSON-encoded strings.
When implementing a function stub, `oldString` is just `raise NotImplementedError(...)` —
a single line with no surrounding context. The model must generate the entire multi-line
function body as the `newString` value inside a JSON string:

```json
"newString": "    sorted_vals = sorted(values)\n    mid = len(sorted_vals) // 2\n    ..."
```

Every line needs explicit 4-space prefixes. Every newline is `\n`. The model silently truncated
the body — the `return sorted_vals[mid]` line was missing from the median implementation.
Tests failed on odd-length lists. The model panicked and re-edited with wrong indentation.

**Why aider doesn't have this problem:** aider uses SEARCH/REPLACE blocks in natural text.
The `SEARCH` section includes the full function signature as context. The model generates the
complete replacement as a code block — natural formatting, no JSON encoding, no indentation risk.

**Fix:** AGENTS.md updated to instruct the model to use `write` (full file rewrite) for stub
implementations. When writing the whole file, the model generates code naturally with correct
context and indentation. The T2 successful run used this approach.

**T3 note:** T3 used `edit` (targeted rename) correctly — `edit` is right for single-line
changes where the `oldString` is a complete, unique identifier. The failure mode only occurs
when `newString` is a multi-line function body.

## Key findings

### 1. Both backends solve all three tiers at 32B

No capability failure at this model size. T1/T2/T3 all pass with correct infrastructure.
To find a failure point, run 7B (predicted failure: T3, missing shapes.py update).

### 2. opencode token cost grows superlinearly with task complexity

The 5.4× T1 ratio becomes 19.7× at T3. This is not model inefficiency — it is a structural
property of full-context history re-transmission. Every turn replays the entire conversation.

**Implication:** opencode's cost model is `O(turns² × file_size)`. For long multi-step tasks,
this compounds fast. A 10-turn task costs disproportionately more than two 5-turn tasks.

### 3. Framework matters more than model for token cost

The same qwen2.5-coder-32b model costs 5.9k tokens via aider and 91k tokens via opencode on
the identical T2 task. The **15.5× gap is entirely framework overhead** — tool schemas
re-transmitted every turn, full history accumulation, Responses API wrapper.

### 4. Tool_choice manipulation is a proxy antipattern

Five iterations of `tool_choice` heuristics were required before landing on the correct
approach (`task_complete` signal). The lesson: any proxy logic that tries to infer task
completion from tool call counts will be fragile and task-specific. Give the model an
explicit exit mechanism.

### 5. aider duration anomaly on T1 (219s vs 122s for opencode)

aider T1 hit a cold-start penalty (219s vs 62s on T2). opencode T1 was 122s both times.
The difference is that opencode re-uses the running vLLM server across runs; aider's duration
included the Modal sandbox cold-start waiting for the vLLM server to warm.
