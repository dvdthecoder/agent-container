# Analysis

Per-run token consumption, cost, and quality comparisons across model × backend combinations.

Each page covers one analysis run: a fixed task fired against all configured backends and models, with results compared side by side.

## What is measured

| Metric | Source |
|---|---|
| Prompt tokens | Tokens sent to the model per turn, accumulated across the run |
| Completion tokens | Tokens generated by the model per turn, accumulated |
| Total tokens | Sum — what gets billed |
| Est. cost | `total_tokens / 1M × $/1M rate` |
| Duration | Wall-clock seconds from sandbox start to container termination |
| Success | Whether the agent produced a non-empty diff and tests passed |

## Why token counts differ between backends

**aider** calls Chat Completions directly. It sends the task, the repo map (~1k tokens), and any edited files. Typical total: **2–5k tokens** per run.

**opencode** calls the Responses API (translated by the in-process proxy). It sends 10 tool schemas (~500 tokens each) on **every turn**, plus accumulated message history. Typical total: **20–50k tokens** per run — the gap is almost entirely framework overhead, not model output.

## Running your own analysis

```bash
# Deploy a model and wait for it to be ready
make deploy MODEL=qwen3-8b

# Run analysis, write a JSON sidecar
make test-analysis \
  MODEL_LABEL="Qwen3 8B · A10G" \
  OUTPUT_JSON=docs/analysis/data/qwen3-8b.json

# Deploy a second model, run again
make deploy MODEL=qwen2.5-coder-32b
make test-analysis \
  MODEL_LABEL="Qwen2.5-Coder 32B · A100" \
  OUTPUT_JSON=docs/analysis/data/qwen32b.json

# Combine into one page
make combine-analysis DATE=$(date +%Y-%m-%d)
```
